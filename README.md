# ResQHub — платформа енергостійкості міста

**ResQHub** моніторить критичні об'єкти міста (укриття, лікарні, пункти незламності, школи, пожежні частини) під час блекаутів і підказує диспетчеру, **куди і що везти в першу чергу**: генератор, батарею, паливо, Starlink чи техбригаду.

> **Демо:** Житомир, Україна · 10 об'єктів · треки: Smart City / AI & Data / Енергетика

---

## 💡 Проблема

Під час масових відключень міські служби не бачать реальний стан сотень укриттів: у кого ще є заряд, у кого працює генератор, куди вже їдуть люди. Рішення ухвалюються телефоном і "на око". ResQHub замінює це живою картою + ML-пріоритизацією.

## 🎯 Що відбувається під час блекауту (ключова логіка)

Кожен об'єкт має **власну ємність батареї** (`battery_capacity_wh`) і **власне навантаження** (тип об'єкта + кількість людей). Тому блекаут для всіх різний:

| Об'єкт | Обладнання | Що станеться |
|---|---|---|
| Лікарня (30 кВт·год, генератор) | АВР | Генератор стартує **автоматично**, працює на пальному 48 год. Втручання не потрібне |
| Пункт незламності (генератор) | АВР | Те саме, запас пального 12 год. Коли пальне закінчиться — з'явиться у рекомендаціях на доставку палива |
| ЗОШ №3 (6 кВт·год, без генератора, 150 людей) | Тільки батарея | Висока витрата → автономність ~1 год → **перший кандидат** на мобільний генератор |
| Укриття ЗОШ №21 (4.5 кВт·год, без генератора) | Тільки батарея | ~2–4 год автономності, деградує поступово, люди переходять у живі укриття |

Модель споживання: `автономність = ємність_батареї × заряд% / (базове_навантаження_типу + 25 Вт × людина)`. Ті самі формули на бекенді (сценарії) і в симуляторі (фізика тіків).

**Семантика полів:** `power_on` = міська мережа, `generator_on` = живлення від генератора. «Об'єкт з живленням» = будь-що з двох.

## 👥 Ролі

| Сторінка | Хто | Що бачить |
|---|---|---|
| `/operations` | Диспетчер | Карта реального часу; на кожному об'єкті — бейдж «що потрібно» (🔌 генератор / 🔋 батарея / ⛽ паливо / 📡 Starlink / 🔧 техбригада); клік → деталі + відправка ресурсу; машинки їдуть по реальних дорогах; живий журнал подій |
| `/analytics` | Міська влада | Середній бал міста по хвилинах, доступність живлення/зв'язку, тренди по районах, повна хронологія подій |
| `/resident` | Мешканець | Найближчі живі пункти незламності: де є світло, зв'язок і вільні місця |

## 🏗️ Архітектура

```

      ▲                                └──────┬───────┘
      │ той самий endpoint                    │
```

Симулятор і реальний мікроконтролер використовують **той самий** `POST /api/telemetry` — залізо підключається без жодної зміни бекенду.

## 🧠 ML / логіка рішень

| Модель / логіка | Навіщо | Де в продукті |
|---|---|---|
| **Resilience Score** — RandomForest (200 дерев) | Бал стійкості 0–100 з кожної телеметрії | Карта, деталі об'єкта, аналітика |
| **Priority Ranker** — LightGBM lambdarank | Пріоритет об'єктів для «під ризиком < 1 год» | KPI на операційній |
| **TTC forecast** — фізична модель | Хвилини до критичного стану (враховує генератор) | Таймери, ескалація статусу |
| **Recommendation engine** | Який ресурс потрібен об'єкту (генератор/батарея/паливо/Starlink/техбригада/евакуація) за станом | Бейдж «Потрібно: X» + кнопка відправки |
| **Модель споживання** — Вт·год | Диференційована автономність кожного об'єкта | Сценарії блекауту + симулятор |

Тренування (потрібно один раз): `python -m app.ml.train` — 8000 семплів, 4 сценарії блекаутів, метрики RMSE/MAE/R²/Brier + calibration plot.

> Додатково у бекенді є `/api/ml/*` ендпоінти для explainability (SHAP), operator briefing, counterfactual, feature drift і online learning + model cards — доступні через API/`/docs`, але навмисно не винесені в основний UI, щоб інтерфейс диспетчера лишався простим.

## ⚡ Швидкий старт

Передумови: Python 3.11+, Node 18+ і pnpm.

```bash
# 1. Backend
cd backend
python -m venv ../.venv && ../.venv/Scripts/activate
pip install -r requirements.txt
cp .env.example .env
alembic upgrade head
python -m app.seed
python -m app.ml.train
uvicorn app.main:app --reload --port 8000

# 2. Simulator (окремий термінал)
cd simulator && ../.venv/Scripts/activate
python main.py --demo          # демо-сценарій для презентації

# 3. Frontend (окремий термінал)
cd frontend && pnpm install && pnpm dev
```

UI: http://localhost:3000 · API docs: http://localhost:8000/docs

## 🎬 Демо-сценарій для журі (3 хвилини)

1. Відкрити `/operations` — все зелене, місто живе звичайним життям, журнал подій порожній.
2. Натиснути **«Симулювати блекаут»** (або `python main.py --demo`).
3. Показати диференціацію: лікарня і пожежна частина **самі** перейшли на генератори (АВР) — стабільні. Школи без генераторів жовтіють/червоніють з різною швидкістю — у кожної свій запас батареї. Об'єкти з живленням, але без зв'язку → **«Увага»** з бейджем 📡 «Потрібно: Starlink».
4. На маркерах видно **що саме везти**: 🔌 генератор малим/великим об'єктам, 🔋 батарея, ⛽ паливо (коли генератор заглух), 📡 Starlink.
5. Клікнути на критичний об'єкт → у деталях натиснути **«Направити генератор»** → машинка їде **по реальних дорогах** до об'єкта → на прибутті об'єкт відновлюється («Допомога в дорозі» → «Стабільно»), у парку стало на 1 генератор менше.
6. Відправити **техбригаду** → вона приїжджає, **«працює» на місці** (🔧), потім **повертається на базу** і знову доступна — на відміну від генератора, що лишається на об'єкті.
7. Переключитись на `/resident` — мешканець бачить, у яких пунктах є світло і вільні місця. На `/analytics` — крива стійкості міста впала на блекауті й відновилась.

## 🔌 Реальне залізо (демо для журі)

Той самий `POST /api/telemetry` приймає дані з реального мікроконтролера в укритті. Мінімальний вузол:

| Компонент | Що вимірює / робить | Орієнтовна ціна |
|---|---|---|
| ESP32 DevKit V1 | контролер, Wi-Fi | ~250 грн |
| MH-Z19B (NDIR CO₂) | CO₂ в укритті | ~900 грн |
| DHT22 / SHT31 | температура + вологість | ~150 грн |
| ZMPT101B | наявність напруги мережі (`power_on`) | ~120 грн |
| INA219 + 18650 + TP4056 | заряд резервної батареї (`battery_pct`) | ~200 грн |
| SIM800L + SIM-картка | канал зв'язку, коли інтернет зник | ~350 грн |
| OLED 0.96" SSD1306 | статус вузла на місці | ~120 грн |
| Макетка, дроти, БЖ | збірка | ~200 грн |

**Разом: ~2300 грн за повний вузол.** Прошивка — цикл із читанням сенсорів і `HTTP POST` JSON кожні 30 с (формат — у `/docs` → `POST /api/telemetry`).

## 📂 Структура

```
backend/    FastAPI: api/, services/ (orchestrator, score, forecast, routing), ml/ (core + monitoring), seed.py
simulator/  asyncio-симулятор з фізикою Вт·год (той самий контракт телеметрії, що й залізо)
frontend/   Next.js 14: operations, analytics, resident
docs/       domain.md — специфікація домену
```

**Деталі реалізації:**
- Доставки завершуються **на бекенді** (авто-completion через час «в дорозі») — відновлення об'єкта не залежить від того, чи вкладка диспетчера у фокусі.
- Машинки рухаються **по реальних дорогах** через OSRM (`router.project-osrm.org`), з fallback на пряму лінію, якщо сервіс недоступний.
- Композитні індекси `(object_id, ts)` на `telemetry`/`scores` → запити «останній стан по кожному об'єкту» ~30 мс замість ~2 с.
- Час скрізь серіалізується як UTC-aware ISO (без зсуву в історії).
